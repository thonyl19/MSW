# MSW 建置系統升級規劃：從 Link.bat 轉向互動式 JS 工具

## 1. 現狀分析 (Link.bat)
目前的 `Link.bat` 是一個靜態的批次檔，其功能與限制如下：
- **功能**：
    - 定義固定的目標路徑 (`R:\MPI_Dev`)。
    - 使用 `mklink /J` 建立 `src` 目錄的連接點 (Junction)。
    - 刪除並重新建立 `mockServiceWorker.js` 的檔案連結。
- **限制**：
    - **硬編碼 (Hardcoded)**：路徑固定為 `R:\` 槽，切換電腦或專案路徑時需要手動編輯批次檔。
    - **權限處理**：`mklink` 需要管理員權限，批次檔無法自動偵測或提示，失敗時錯誤訊息不明顯。
    - **互動性差**：無法在建置前選擇模式（例如：連結開發模式 vs. 直接複製模式）。

## 2. 規劃方案：互動式 `setup-link.js`
建議將此邏輯整合至 Node.js 環境中，建立一個互動式的設置工具。

### A. 整合方式
- 在 `package.json` 中新增腳本：`"setup": "node scripts/setup-link.js"`。
- 讓開發者只需執行 `npm run setup` 就能完成環境配置。

### B. 功能設計
1. **環境檢查階段**：
   - 檢查是否具備管理員權限（`mklink` 必要）。
   - 檢查源檔案（`src` 目錄與 `mockServiceWorker.js`）是否存在。
2. **互動配置階段**：
   - 使用 `readline` 或 `inquirer` 詢問目標專案根目錄, 只需要提供目錄和專案名稱即可,例如:`R:\MPI`,程式自動加上後綴  `_Dev` , 最後生成的路徑為`R:\MPI_Dev`。 
   - 提供開發模式選擇：
     - **Link Mode (推薦)**：使用 `junction/link`，本地修改立即反映至專案。
     - **Sync Mode**：直接執行實體檔案複製（適合無法使用連結的環境）。
3. **執行階段**：
   - **自動清理**：偵測目標路徑是否已存在舊的連結或資料夾，並詢問是否覆蓋。
   - **建立連結**：使用 Node.js `fs.symlinkSync` 完成跨目錄對接。
   - **設定持久化**：將選擇的路徑存入本地 `.env` 或 `config.json`，下次執行時免輸入。

## 3. 預期效益
- **靈活性**：不再受限於 `R:\` 槽，可適用於任何專案位置。
- **安全性**：在刪除或覆蓋目標檔案前進行確認，減少誤刪風險。
- **整合性**：可與現有的 `build.js` (esbuild) 流程整合，實現一鍵完成「連結 + 編譯」的完整流程。
- **視覺化反饋**：使用顏色與圖示提示建置結果，方便排錯。
