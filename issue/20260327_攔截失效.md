# 2026-03-27 攔截失效問題分析報告 (Issue Report)

## 1. 現象描述
執行「熱重載 (Hot Reload)」後，雖然 Console 顯示重載完成，但原本模擬的 API 全數導向後端伺服器 (Interception failed)，且瀏覽器 Console 出現兩次啟動訊息：
- `[MSW] 攔截系統已啟動 (作用域: /MPIMES_DEV/)` (初始)
- `[MSW] 攔截系統已啟動 (作用域: /MPIMES_DEV/)` (熱重載後再次出現)

## 2. 根源分析 (Root Cause)
### ESM 模組單例衝突 (Module Singleton Violation)
1. **快取標記過度擴張**: 在 `reloadAllMocks` 的原始實作中，為了強制刷新，對 pages 內的所有引用（如 `import { registerMock } from '../mock-entry.js'`) 都加上了 `?t=TIMESTAMP`。
2. **分身實例產生**: 當模組路徑帶上 `?t=` 時，瀏覽器會將其視為「全新的模組 URL」，因此載入了第二個 `mock-entry.js` 實例。
3. **管理權轉移且脫鉤**: 
   - 頁面模組 (Blob) 執行時呼叫的是 **「分身 `mock-entry.js`」** 內的 `registerMock` 方法。
   - 分身實例內部的 `worker` 與 `startWorker()` 與最初跟 Service Worker 綁定的「本尊實例」完全不同。
   - 導致攔截邏輯被註冊到了「沒在作用的 Worker 實例」上，而正處於攔截狀態的「本尊實例」則處於 `resetHandlers()` 後的清空狀態。

## 3. 解決方案方案 (Fix Action)
### 實作：核心檔案排除清單 (White-list Policy)
- **規則**: 在動態源碼解析階段，過濾掉 `store.js`, `mock-entry.js`, `msw-utils.js`, `handlers.js` 等核心組件及其依賴。
- **效果**: 
  - 這些檔案在 import 時不帶 `?t=`，瀏覽器會直接從記憶體快取返回 **最初的本尊實例**。
  - `registerMock` 呼叫後能正確更新正在攔截中的 `worker` 實例。
  - 僅針對頁面資料與邏輯進行 Cache Bursting。

## 4. 修正記錄
- 檔案：`src/mock-entry.js`
- 變更：優化 Regex 注入邏輯，加入檔案名稱比對與排除機制。
