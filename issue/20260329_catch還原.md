# 議題分析與技術總結：MSW Catch 狀態還原失效修復

## 1. 問題描述 (Issue Summary)
使用者反映在 MSW 面板點擊 `💾 (Catch)` 快照後，重新整理頁面 (Reload)，發生以下情形：
- 面板中的控制項（如 Select 下拉選單）未正確還原至快照儲存的數值。
- 主畫面初次加载時執行的程序，未根據快照設定的 Mock 狀態正確運行（Injection 沒帶入）。

## 2. 核心根因分析 (Root Cause Analysis)

### 2.1 時序問題 (Timing Issue)
原本的快照還原邏輯放在 `MockPanel.js` 的 `setTimeout(..., 500)` 中。這導致在頁面 Reload 時，主畫面的開發環境 (如 Vue / API 請求) 往往在幾十毫秒內就發送了請求。
這時：
1. `_registerPage` 剛執行完畢，數據尚未還原（`undefined`）。
2. MSW Handler 獲取的是初始值。
3. 等到 500ms 後 `loadCatch` 執行時，主程序的初次初始化已結束。

### 2.2 狀態綁定不一致 (Binding Mismatch)
面板 UI (Vue Templat) 透過 `v-model="config[key]"` 直接綁定至 `mockConfig` 的全局屬性。
然而，原本的 `saveCatch` 邏輯是存儲 `control.value` 對象內容，但在 UI 操作時並未同步更新該對象，導致存入快照的是舊值或無效值。

## 3. 關鍵修正程序 (Key Fixes)

### 3.1 主動式還原 (Proactive Restore)
將快照還原的時機從 UI 元件掛載時，提前至 **`store.js` 的 `_registerPage` (註冊階段)**。
- 當頁面腳本呼叫 `registerMock` 時，系統立即檢查 `sessionStorage` 是否有當前路徑的快照。
- 若有，則在建立響應式屬性的瞬間，直接將快照值賦予全局狀態。
- **優點**：確保後續所有 API 攔截與 UI 渲染從一開始就讀到正確的快照數據。

### 3.2 狀態同步邏輯 (Global State Sync)
- **儲存時**：由 `saveCatch` 直接從 `mockConfig` 全局對象中抓取所有已註冊 Key 的當前數值。
- **還原時**：利用 `Vue.set` 確保全局屬性存在且具備響應式，保證 `select` 等元件能確實更新。

### 3.3 持久化最後執行動作 (Last Action Persistence)
- 快照中新增 `lastAction` 欄位，記錄最後一次 Injection (Inject 分頁) 的目標與數據。
- 在 `MockPanel.js` 啟動後，若偵測到快照中有 `lastAction`，主動執行一次 `triggerAction`。
- **效果**：解決了「Reload 後數據沒帶入主畫面」的問題。

## 4. 維護建議與注意事項

- **Key 的唯一性**：由於 `mockConfig` 是扁平的全局對象，不同頁面若使用相同的 `key` 名稱會發生覆寫。建議在控制項定義時，對於跨頁面不同的數值使用具備前綴的 Key。
- **不可隨意更換 State 結構**：若未來修改 `store.js` 中的 `mockConfig` 結構，必須同步檢查 `saveCatch` 與 `loadCatch` 的路徑映射。
- **清除快照**：若因快照格式錯誤導致還原失敗，請使用面板頂部的 `🧹 (Clear Catch)` 按鈕重置。
