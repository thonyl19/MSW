# 技術規範：直接注入模式 (Direct Injection Mode)

## 0. 背景與目的
傳統的注入機制依賴主頁面呼叫 `useFormInjection` 並主動訂閱 `mockConfig.lastAction`。為了達成「零改動、強兼容」的測試體驗，我們開發了「直接注入模式」，讓 MSW 面板能主動掃描 DOM 並操縱 Vue 實例。

## 1. 核心實作重點 (Implementation Highlights)

### A. 主動追蹤路徑
不依賴事件匯流排，而是主動從以下根節點進行深搜 (Recursive Search)：
- `#app`
- `.app-container`
- `body > div` (通用掛載點)

### B. 遞迴實體搜尋邏輯
透過 `el.__vue__` 屬性進入 Vue 的狀態樹，並使用遞迴搜尋 `$children`，直到找到具備「目標屬性名稱」(例如 `form`) 的組件實例為止。

### C. 響應性安全性
由於 Vue 2 的限制，注入過程必須遵循以下規則：
1. **使用 `this.$set` (Vue.set)**：確保新注入的資料屬性能被 Vue 監聽到並反應回 UI。
2. **強制 `_.cloneDeep`**：確保 Mock 資料來源 (Data Source) 是唯讀的，不會被頁面組件的雙向綁定所污染。

## 2. 變更禁區 (Guiding Principles - DO NOT BREAK)

### 1. 邏輯隔離原則
**嚴禁將注入資料 (Form Injection) 寫入 `mockConfig.activePayload`。**
- `activePayload` 是全域 API 攔截回應的覆寫機制 (Data Stress)。
- 注入資料僅限於操作「組件 Data」，不得干擾「API Response」，否則會導致 API 返回值異常。

### 2. 目標明確原則
注入動作必須包含 `target` 參數。如果 `control.target` 為空，應預設為 `form`。若搜尋不到具備該屬性的實體，應在 Console 提供黃色告警 (`console.warn`) 而非報錯中斷。

### 3. 持久化副作用
注入屬於「執行時一次性動作」，不應產生持久化的全域狀態改變（除 `lastAction` 紀錄外），以確保測試環境的可回溯性。

## 3. 關鍵程式碼位置
- 觸發點：`src/components/MockPanel.js` -> `triggerAction()`
- 注入點：`src/components/MockPanel.js` 內部的 `this.$nextTick` 區塊
